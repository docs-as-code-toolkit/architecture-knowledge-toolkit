= ADR-002: Defer a Public API Until External Consumers Are Confirmed

== Status

Proposed.

This ADR is an AI-assisted draft. It must not be treated as accepted architecture
truth until a human reviewer confirms the decision and updates the metadata.

== Context

The sample project models an application that publishes domain events through a
Kafka event backbone. The existing architecture proposal focuses on asynchronous
event delivery, independent consumer scaling, and replay for selected recovery
cases.

The open question is whether the application should also provide a public API
for external consumers.

A public API would give consumers a direct integration surface. It would also
create a compatibility promise for request and response contracts, error
semantics, authentication, authorization, rate limits, observability, support,
and versioning.

The sample project does not yet record concrete external consumers or use cases
that require direct synchronous access to application capabilities. Without that
evidence, a public API would be an assumed product requirement rather than a
reviewed architecture need.

== Decision Drivers

* The sample architecture should keep integration contracts explicit and
  reviewable.
* The team should avoid freezing public contracts before external consumer needs
  are known.
* The existing event backbone should remain the preferred integration mechanism
  for asynchronous domain events.
* Any public API needs clear ownership for security, versioning, compatibility,
  documentation, monitoring, and support.
* The application should not expose domain entities directly at a layer
  boundary.

== Considered Options

=== No Public API

The application does not expose a public API. External integration happens only
through reviewed event contracts, internal interfaces, or manual channels.

This option keeps the architecture small and avoids a premature compatibility
promise. It gives external consumers no direct application access.

=== Public API Now

The application exposes a public API as a first-class integration surface.
External consumers can call application capabilities directly through stable
contracts.

This option improves direct integration, but it creates product, security,
versioning, documentation, and operational obligations before consumers and use
cases have been confirmed.

=== Deferred Public API

The application does not expose a public API now. The team revisits the decision
when concrete external consumers, use cases, quality scenarios, and ownership
rules are recorded.

This option preserves the option to expose an API later while keeping the
current architecture focused on confirmed integration needs.

== Decision

We propose to defer a public API until concrete external consumers are
confirmed.

For now, the sample application should not expose a public synchronous API as an
architecture commitment. The reviewed integration surface remains the proposed
event backbone and any internal contracts needed by the implementation.

A public API may be introduced later when the team records consumer use cases,
request and response contracts, authentication and authorization needs,
compatibility rules, rate limits, monitoring, and support ownership.

== Consequences

=== Positive

* The application avoids a public compatibility promise before consumer needs
  are known.
* The current architecture can focus on the event-driven integration model.
* Security and operations scope stays smaller for the next increment.
* A later API can be designed from reviewed use cases instead of assumptions.

=== Negative

* External consumers cannot integrate directly through synchronous API calls
  until the decision is revisited.
* Some consumers may need temporary manual, internal, or event-based
  integration paths.
* Future API introduction may require additional architecture work for
  authentication, authorization, versioning, rate limiting, and observability.

=== Neutral or Follow-Up

* The team should capture at least one external consumer use case before
  revisiting the decision.
* The team should define API DTOs explicitly if a public API is introduced.
* The team should create quality scenarios for API latency, availability, and
  security before accepting a future API decision.

== Assumptions

* No committed stakeholder requirement currently demands a public API.
* The event backbone is sufficient for the integration scenarios currently
  represented in the sample project.
* Internal application interfaces, if any, are not public compatibility
  contracts.

== Open Questions

* Which external consumer would use the API first?
* Which user goal or system use case requires synchronous access?
* Which security model would apply to public API clients?
* Which API quality scenarios must be met before the API can be accepted?

== Review Notes

Before this ADR can be accepted, reviewers should confirm whether external
consumers exist, whether event-based integration is sufficient for the next
increment, and who would own a public API if one becomes necessary.
