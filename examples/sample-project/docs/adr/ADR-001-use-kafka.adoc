= ADR-001: Use Kafka as the Event Backbone

== Status

Proposed.

This ADR is an AI-assisted draft. It must not be treated as accepted architecture
truth until a human reviewer confirms the decision and updates the metadata.

== Context

The sample system publishes domain events from multiple services. Consumers need
asynchronous delivery, replay for selected event streams, and independent scaling
for high-volume processing.

The architecture should support event-driven integration without coupling service
release cycles too tightly.

== Decision Drivers

* Event processing latency must stay predictable under normal business load.
* Consumers should be able to scale independently.
* Event history should be available for replay during operational recovery.
* Teams need clear ownership for event contracts and platform operations.

== Considered Options

=== Direct Synchronous APIs

Synchronous APIs are simple to understand, but they couple producers and
consumers at runtime and make replay difficult.

=== Database Polling

Polling existing databases avoids introducing a platform component, but it leaks
internal persistence models and makes ownership unclear.

=== Kafka Event Backbone

Kafka provides durable event streams, consumer groups, partitions, replay, and a
well-understood operational model for asynchronous event processing.

== Decision

We propose to use Kafka as the shared event backbone for asynchronous domain
events in the sample architecture.

== Consequences

=== Positive

* Consumers can scale independently through consumer groups.
* Event streams can be replayed for selected recovery and reprocessing cases.
* Event delivery and retention become explicit platform capabilities.

=== Negative

* Kafka becomes a central operational dependency.
* Event schema governance is required to avoid accidental coupling.
* Teams need operational knowledge for partitions, retention, and consumer lag.

=== Neutral or Follow-Up

* The team must confirm whether Kafka operational complexity is acceptable for
  the sample context.
* The proposed risk `RISK-001-event-backbone-coupling` must stay visible while
  the decision is under review.

* Which team owns Kafka topics, retention settings, and schema compatibility?
* Which event streams need replay, and for how long?
* Which monitoring signal proves that the latency scenario is met?
