[[qs-001-event-processing-latency]]
= QS-001: Event Processing Latency Under Normal Load

== Status

Proposed.

The response measure is intentionally concrete so the scenario can be reviewed
and tested. The value remains an assumption until confirmed by the accountable
owner.

== Scenario

[cols="1,3", options="header"]
|===
| Element | Description

| Source
| A domain service publishes a business event.

| Stimulus
| A valid event is appended to the event backbone during normal business load.

| Environment
| Normal operation, with expected daily traffic and no active platform incident.

| Artifact
| Event backbone and subscribed consumer services.

| Response
| Subscribed consumers can receive and start processing the event.

| Response Measure
| 95% of valid events are available to active consumer groups within 2 seconds of publication.
|===

== Rationale

Several user-facing workflows depend on timely propagation of domain events. The
latency target is intentionally modest for the sample project, but concrete
enough to drive architecture decisions and operational monitoring.

== Assumptions

* Normal business load is known and can be expressed as an event throughput
  range during review.
* Active consumer groups are healthy and not intentionally paused.
* Publication time and consumer availability time can be measured from platform
  telemetry.
