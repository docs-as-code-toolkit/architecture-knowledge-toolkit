[[adr-007-artifact-provenance-metadata]]
== ADR-007: Architecture Artifact Provenance Metadata

=== Status

Proposed.

This ADR is AI-assisted and must not be treated as accepted architecture truth
until reviewed and accepted by the accountable owner.

=== Decision

We propose to capture optional artifact provenance in `derived_from` metadata
and expose selected metadata values through generated per-artifact AsciiDoc
attribute fragments. `derived_from` records why an artifact was created, while
`relations` continue to model architecture traceability between artifacts.

=== Context

During exploratory testing, an initial architecture question was answered in
chat without changing documentation. A follow-up prompt then asked the agent to
treat that answer as a decided architecture decision. The resulting ADR and
Q&A entry were useful, but the ADR did not preserve the originating question as
machine-readable provenance.

The existing relation model intentionally connects architecture artifacts by
stable IDs. That works well for traceability between ADRs, quality scenarios,
risks, concepts, and documents. It does not fit every origin: an ADR may be
derived from a prompt, a Q&A anchor, a repository path, an external URL, or a
short source note that is not itself an architecture artifact.

AsciiDoc source bodies cannot access YAML front matter directly. To make
metadata visible in prose without duplicating it manually, the generator creates
small includeable attribute fragments such as
`generated/adr-007-artifact-provenance-metadata-attributes.adoc`.

=== Decision Drivers

* Preserve the origin of AI-assisted and human-authored architecture artifacts.
* Avoid overloading architecture relations with prompt or source provenance.
* Keep provenance optional so lightweight projects are not forced to model every
  input as a first-class artifact.
* Make generated indexes and source prose able to show provenance when
  available.

=== Considered Options

==== Option 1: Optional `derived_from` metadata with generated attributes

Add a small optional metadata field that accepts text entries or structured
origin entries with a description, URL, repository path, AsciiDoc anchor, or
artifact ID. Generate AsciiDoc attributes from selected metadata when source
prose needs to render model values.

==== Option 2: New `derived_from` relation type

Add provenance as another relationship type in the relation metamodel.

==== Option 3: Prose-only origin notes

Keep provenance only in ADR context sections or review notes.

=== Consequences

==== Positive

* ADR indexes can show where a decision came from when provenance is recorded.
* ADR source text can render provenance through generated attributes without
  duplicating metadata by hand.
* Agents can link an ADR back to an originating Q&A anchor or external source
  without inventing a traceability relation.
* Reviewers can inspect the input context that caused an architecture artifact
  to exist.

==== Negative

* The metamodel gains one more optional metadata field.
* Source files that render metadata values depend on generated attribute
  fragments being present before rendering.
* Authors must still decide whether an origin is only provenance or also an
  architecture relation.

==== Neutral or Follow-Up

* Provenance remains optional and does not change artifact lifecycle status.
* Existing artifacts do not need backfilled `derived_from` metadata unless their
  origin is useful for review.

=== Assumptions

* Reviewers benefit from seeing an artifact's origin in generated indexes.
* A provenance field with optional structured links is enough for the current
  generator and does not require first-class prompt artifacts.
* Generated attribute fragments are an acceptable way to bridge YAML metadata
  into AsciiDoc prose.

=== Open Questions

* Should provenance also be shown in per-artifact generated traceability
  fragments, or is the ADR index sufficient for the first slice?
