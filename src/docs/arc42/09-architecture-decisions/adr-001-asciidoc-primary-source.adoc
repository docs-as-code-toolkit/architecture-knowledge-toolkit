== ADR-001: Primary Architecture and Documentation Source Format

=== Status

Proposed.

This ADR is AI-assisted and must not be treated as accepted architecture truth
until reviewed and accepted by the accountable owner.

=== Decision

We propose to use AsciiDoc as the primary architecture and project-owned
documentation source format.

=== Context

The toolkit follows a Docs-as-Code approach and uses arc42 documentation under
`src/docs/`. The current technology constraints already name AsciiDoc as the
primary documentation source. The repository also uses PlantUML and
`include::[]` for reusable source fragments. This also applies to
project-owned presentation material: slide decks and supporting talk artifacts
should use AsciiDoc when they are maintained as repository documentation.

=== Decision Drivers

* Support arc42 documentation and local static rendering.
* Support reusable source fragments without duplicating source text.
* Keep source files reviewable in pull requests.

=== Considered Options

==== Option 1: AsciiDoc as primary source

AsciiDoc fits the current repository structure, supports includes, and matches
the Docs-as-Code Toolkit ecosystem. It also supports reveal.js decks and
reusable talk fragments through the same source format.

==== Option 2: Markdown as primary source

Markdown is familiar to many contributors, but it gives weaker native support
for structured includes and arc42-style publishing.

==== Option 3: Mixed source formats

Mixed formats may help some contributors, but they increase validation,
generation, and style complexity.

=== Consequences

==== Positive

* Architecture and project-owned documentation content can use native AsciiDoc
  includes for shared fragments.
* The publication path can start with plain Asciidoctor-generated HTML.

==== Negative

* Contributors need basic AsciiDoc knowledge.
* Reusable fragments create rendering risk if they assume a fixed heading level
  or target format. This is tracked by xref:r-003-source-fragments-break-rendering[].

==== Neutral or Follow-Up

* Markdown may still be used where it is already the platform standard, such as
  repository README files, GitHub issue bodies, and other external surfaces.
