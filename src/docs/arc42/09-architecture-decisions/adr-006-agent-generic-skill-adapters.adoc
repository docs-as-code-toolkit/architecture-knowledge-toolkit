[[adr-006-agent-generic-skill-adapters]]
== ADR-006: Use canonical engine-independent skills with generated agent adapters

=== Status

Proposed.

This ADR is AI-assisted and must not be treated as accepted architecture truth
until reviewed and accepted by the accountable owner.

=== Decision

We maintain architecture skills once under `skills/**/SKILL.md` and generate or
derive agent-specific adapters from them. Agent-specific files must remain thin
wrappers and must not duplicate architecture semantics.

=== Context

The toolkit already proposes an engine-specific runtime integration boundary in
xref:adr-005-engine-independent-skills[ADR-005]. The remaining problem is drift:
as Codex, Vibe, GitHub Copilot, Cursor, OpenAI, and future agent integrations
grow, adapter files can accidentally become competing sources for ADR,
quality-scenario, risk, traceability, metadata, or arc42 rules.

`Green-PT/honey-for-devs` is a useful structural reference because it authors a
canonical skill once, derives per-agent rule files, and provides a check mode to
detect drift. This decision adopts that structural pattern only. It does not
adopt Honey token-optimization behavior, ESON, CCR, PX, automatic compression,
or Honey-specific runtime commands.

=== Decision Drivers

* Preserve one reviewed source for architecture semantics.
* Make agent-specific integrations easy to regenerate and review.
* Reduce drift between adapters and canonical skills.
* Keep Honey-specific runtime behavior out of this toolkit unless future work
  explicitly accepts it.

=== Considered Options

==== Option 1: Canonical skills with generated thin adapters

Keep architecture semantics in `skills/**/SKILL.md` and
`general-semantic-contracts.md`. Generate adapter wrappers under `adapters/**`
that point agents back to those canonical sources.

==== Option 2: Hand-maintained thin adapters

Keep adapters manually maintained and rely on contributor discipline to avoid
semantic drift.

==== Option 3: Agent-specific skill copies

Create a separate skill set for each agent and keep the behavior explicit in
each runtime-specific location.

=== Consequences

==== Positive

* Architecture semantics remain concentrated in canonical skills and general
  contracts.
* Generated wrappers make adapter intent visible without duplicating rules.
* CI can fail when generated adapters drift.

==== Negative

* Contributors must run the adapter generator after changing canonical skills.
* Very specific agent integrations may still need hand-written adapter files
  beside the generated wrappers.

==== Neutral or Follow-Up

* The initial generator should stay intentionally small and dependency-free.
* OpenAI-specific skill UI metadata remains under `adapters/openai/<skill-name>/`
  and can be generated later if the metadata format stabilizes.

=== Assumptions

* Adapter files can remain useful while pointing to canonical skills instead of
  restating their rules.
* A small generator is sufficient for current Codex, Vibe, GitHub Copilot, and
  Cursor wrappers.

=== Open Questions

* Which OpenAI adapter metadata fields are stable enough to generate without
  obscuring useful hand-authored metadata?

=== Review Notes

Confirm that generated wrappers are acceptable for each supported agent before
marking this ADR reviewed or accepted.
