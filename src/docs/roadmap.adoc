This roadmap is a suggested new-project roadmap. It uses the existing roadmap as
a hint and incorporates reviewed workshop answers. Derived content still needs
review before it becomes accepted project truth.

== Milestone 0: Project Inception

Goal: make the project purpose discussable and reviewable.

* Create Business Model Canvas and Value Proposition Canvas.
* Draft vision and mission.
* Capture architecture, communication, and techstack canvases.
* Record open questions by role.
* Confirm individual architects as the first target user segment.
* Confirm the initial success metric for guided arc42 Docs-as-Code setup.
* Confirm project setup only as the first adoption slice.

Exit criteria:

* Product owner answers the first target-user, success-metric, and adoption
  boundary questions.
* Maintainers agree which assumptions can become reviewed project facts.

== Milestone 1: Repository Foundation

Goal: make the toolkit structure clear and usable for an individual architect
starting a project.

* Stabilize repository layout.
* Keep reviewed source content separate from generated output.
* Provide a guided initial setup path for arc42 Docs-as-Code documentation.
* Include inception canvases, vision and mission, roadmap, and open-question
  capture in the setup path.
* Make cross-reference conventions visible in templates and examples.
* Make the setup workflow useful on its own before later slices are adopted.
* Document contribution workflow.
* Keep engine-specific integration under `adapters/`.
* Maintain examples that demonstrate expected usage.

Exit criteria:

* An individual architect can create an initial project documentation skeleton
  with inception canvases, vision and mission, roadmap, and open questions.
* The README points to the new-project workflow.

== Milestone 2: Metamodel Validation

Goal: make architecture metadata trustworthy.

* Use AsciiDoc front matter as the metadata location for now.
* Validate artifact metadata against `metamodel/artifact.schema.yaml`.
* Validate relation metadata against `metamodel/relations.schema.yaml`.
* Validate cross references that are modeled as explicit relationships.
* Enforce explicit relationships across ADRs, quality scenarios, and risks.
* Enforce explicit component relationships to ADRs, quality scenarios, and
  risks.
* Document stable ID conventions.
* Add focused tests for valid and invalid metadata.

Exit criteria:

* Broken metadata fails validation with actionable messages.
* Valid sample artifacts pass without manual setup.

== Milestone 3: Deterministic Views

Goal: make reviewed metadata useful in generated documentation.

* Generate a traceability matrix from explicit relations.
* Generate a relationship view for any given artifact.
* Generate artifact-type indexes with links to individual artifacts.
* Revisit later whether additional indexes by status, owner, and tag are worth
  adding in version 1.
* Generate includable documentation fragments for CI/CD pipelines.
* Add regression tests for generator output.
* Ensure generated files are ignored by artifact scanning.
* Keep local and CI generation behavior reproducible in GitHub Actions and
  GitLab CI.

Exit criteria:

* Re-running the generator without input changes produces no meaningful diff.
* Generated views never become the primary editing surface.
* Generated fragments can be included in published architecture documentation.

== Milestone 4: Skill And Adapter Hardening

Goal: make AI assistance reusable and auditable.

* Harden first-class skills for ADRs, quality scenarios, and risks.
* Prepare next-priority skills for stakeholders, requirements, and components.
* Add skills that suggest architecture documentation updates during feature
  implementation and refactoring.
* Refine reusable skills through real project use.
* Add adapter-specific invocation examples.
* Document how suggestions become reviewed source changes.
* Automate publishing of generated documentation to a team-accessible target.
* Keep engine-specific assumptions out of core skills.

Exit criteria:

* A user can tell which content is suggested, generated, or reviewed.
* Adapter guidance remains thin and testable.

== Open Roadmap Decisions

* Which publishing target should be supported first?
* Which implementation language should be used for future generators?
* Which SDLC skills are required first: feature implementation, refactoring, or
  architecture review?
