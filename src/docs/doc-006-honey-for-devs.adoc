= Honey for Devs Reference Patterns

This page records the structural patterns from
https://github.com/Green-PT/honey-for-devs[Green-PT/honey-for-devs] that are
relevant to architecture-knowledge-toolkit.

Honey itself is not an architecture semantics source for this toolkit. Do not
copy Honey token-optimization rules, ESON, CCR, PX, auto-intensity behavior, or
runtime compression behavior into architecture-knowledge-toolkit unless a
future ADR explicitly accepts that scope.

== Relevant Patterns

=== Canonical SKILL.md as single source of truth

Honey authors its core skill once under `skills/honey/SKILL.md` and derives
agent-facing rule files from that source. The matching toolkit pattern is to
keep architecture semantics in `skills/**/SKILL.md` and
`general-semantic-contracts.md`.

=== Generated adapter/rule files

Honey publishes per-agent rule files for tools such as Codex, Cursor, GitHub
Copilot, Cline, and others. The matching toolkit pattern is to generate thin
files under `adapters/**` that point back to canonical skills and contracts.
Canonical skills that should not appear in generated adapters can opt out with
`adapter_expose: false` in their `SKILL.md` front matter.

=== CI drift check

Honey documents a `--check` mode that fails when generated rule files drift
from the canonical skill. The matching toolkit pattern is `./build.sh
check-adapters` (locally `node scripts/check-agent-adapters.js`), suitable for
CI.

=== Core skill plus satellite skills

Honey separates an always-on core from on-demand satellite skills. The matching
toolkit pattern is `skills/architecture-core/SKILL.md` as the broad entry point,
with focused skills such as `skills/adr/SKILL.md`,
`skills/quality-scenario/SKILL.md`, `skills/risk/SKILL.md`,
`skills/slice-issues/SKILL.md`, and `skills/traceability-review/SKILL.md`.

=== Safety carve-outs

Honey explicitly names cases where compression should not apply. The matching
toolkit pattern is to preserve safety boundaries for architecture content:
do not compress away ADR rationale, quality scenario measures, risk context,
stakeholder requirements, security directives, migration directives, or other
review-critical content.

== Out Of Scope

The following Honey-specific behaviors are not adopted here:

* ESON, CCR, PX, or Honey wire formats.
* Token accounting, CO₂ accounting, or cost estimation as architecture
  semantics.
* Automatic prompt compression or output-minimization rules.
* Honey runtime commands, hooks, badges, installers, or subagent behavior.

These may be referenced only as optional external operational capabilities when
the user explicitly asks for GreenIT-aware agent work.
