= Architecture Communication Canvas
:id: CANVAS-004-architecture-communication
:status: draft
:owner: project maintainers
:revdate: 2026-06-27

This canvas describes how the architecture knowledge should be communicated
during project inception and review.

== Audiences

* Maintainers need to understand what belongs in the toolkit core.
* Individual architects are the first audience for setup guidance, as confirmed
  by xref:q-po-001[Q-PO-001].
* Architects need to apply the templates and metadata rules consistently.
* Developers need to understand how validation affects contributions.
* Reviewers need to separate suggested content from reviewed content.
* Adapter authors need to keep runtime-specific behavior isolated.

== Communication Goals

* Make the source of truth explicit.
* Show how to create, validate, review, and generate architecture knowledge.
* Explain how reusable documentation fragments support writing once and
  publishing everywhere.
* Explain which content is authoritative and which content is suggested.
* Make traceability useful without turning documentation into ceremony.

== Key Messages

* Repository source files become authoritative only through human review.
* AI may suggest content, but it does not accept decisions.
* Metadata carries traceability; prose alone is not enough.
* Content that belongs in several places should be maintained once as a
  reusable AsciiDoc source fragment.
* Generated views are derived artifacts, not the primary editing surface.
* Generated documentation is regenerated locally or in CI and is not committed
  to the repository, as confirmed by xref:q-arch-005[Q-ARCH-005].
* Deterministic output is a quality requirement.

== Channels And Artifacts

* `README.md` for orientation, starting with the individual-architect setup
  path.
* `src/docs/canvas/` for inception canvases and workshop questions.
* `src/docs/` for reviewed inception documentation and future reusable source
  fragments.
* `src/docs/doc-001-arc42.adoc` and `src/docs/arc42/` as the living dogfood example.
* `skills/` for reusable AI-assisted workflows.
* `scripts/validate-metamodel.rb` for executable validation.

== Feedback Loops

* Pull request review for source artifacts and metadata.
* Validator test failures for schema and generator regressions.
* Workshop answers captured in xref:canvas-questions-and-answers[].
* Open questions reviewed before moving artifacts from draft to reviewed.
